# TIN-18 — M2.4 Tauri command `start_opencode`

## Shipped

New Tauri command `start_opencode(folder_path, user_id, memory_subdir) -> OpencodeHandle { baseUrl, pid }` in `apps/desktop/src-tauri/src/commands/opencode.rs`. Spawns `opencode serve --cwd <folder>` with env `SMART_VAULT_PATH=<memory_subdir>` plus per-spawn basic-auth credentials. Health-polls `GET /health` every 100ms, 10s timeout. Writes `~/.tinker/manifests/<session-id>.json` (0600 on unix) with `{pid, port, secret, folder_path, user_id, memory_subdir, base_url, session_id}`. Detaches `CommandChild` into a drain task so the process survives app quit (shared-child drop is non-signalling).

Also renamed the legacy internal `start_opencode` fn in `lib.rs` to `bootstrap_opencode` to free the command-name slot. No behavior change on the bootstrap path.

## Reasoning

Per D22 (no mutate-then-call): config is passed per invocation. Per D17 (coordinator pattern): child handle is dropped, state lives in the manifest file for future adoption across app restarts.

Elon-reviewer flagged four refinements which were applied:

1. Dropped `sessionId` from `OpencodeHandle` — ticket only promised `{baseUrl, pid}`; session_id already lives in the manifest filename + body.
2. Added `0o600` on the manifest file — it stores a cleartext basic-auth secret; default umask on multi-user unix would have been 0644.
3. Deleted `TINKER_USER_ID` / `TINKER_SESSION_ID` envs — nothing reads them.
4. Inlined `port_from_base_url` + dropped its two tests — single caller.

Reviewer also flagged a BLOCKING scope concern: the app-boot `bootstrap_opencode()` still exists and kills its child on exit, contradicting the D17 unref invariant. Per "no scope creep" that surgery is filed as TIN-108 instead of inlined here. The two lifecycle models coexist for now; TIN-108 deletes the global one once the folder-scoped command has at least one caller.
