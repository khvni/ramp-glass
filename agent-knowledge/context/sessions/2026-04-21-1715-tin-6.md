## What shipped

- `apps/desktop/src/renderer/workspace/pane-registry.ts` — added module-level typed registry (`registerPane<K>`, `getRenderer<K>`, `resetPaneRegistry`) on top of the existing Dockview-coupled surface. Renderers narrow per kind via `Extract<TinkerPaneData, { kind: K }>`.
- `apps/desktop/src/renderer/workspace/pane-registry.test.ts` — 6 runtime cases: per-kind registration, payload narrowing, all four kinds, missing-kind error message, duplicate-registration error, reset behavior.
- `apps/desktop/src/renderer/workspace/pane-registry.test-d.ts` — compile-time assertions using `@ts-expect-error` guards (mirrors `shared-types/pane.test-d.ts`).

## Decisions / notes for future work

- **Legacy Dockview surface kept in place.** `createPaneRegistry` + `WorkspacePaneMap` still export from the same file because `Workspace.tsx` imports them. Removing them is M1.7/M1.8 scope — touching `Workspace.tsx` from a task labelled `S / 1.1 → 1.2` would have widened the PR beyond its slice (tasks.md atomicity rule).
- **Generic write-at-mapped-type cast.** TS can't prove `renderers[kind] = render` is sound when `kind: K` is generic. Contained the cast to a single line (`renderers as Record<TinkerPaneKind, unknown>`) at the write site — readers still see exact `PaneRenderer<K>` via the mapped type, so the unsafe region is one line rather than the public surface.
- **Duplicate-registration throws** rather than silently overwriting. Matches the "populated once at app boot" acceptance bullet and surfaces wiring bugs immediately. `resetPaneRegistry` exists as a test-only escape hatch.
- **No `buildPaneRegistry()` adapter shipped.** M1.7 will write the thin adapter that converts the module map into `PaneRegistry<TinkerPaneData>` consumable by `@tinker/panes`'s `<Workspace>`. Keeping that out of M1.2 maintained the atomic slice.
